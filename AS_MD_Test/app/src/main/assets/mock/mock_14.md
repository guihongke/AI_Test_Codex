# 长文本会话 14：售后工单摘要

售后工单的难点在于信息来源分散：用户描述、客服补充、物流节点、商品信息和历史处理记录经常混在一起。AI 摘要需要先把事实和判断分开，再给出可执行建议。事实部分应该保持客观，例如“用户在 6 月 2 日反馈外包装破损”；判断部分可以说明“疑似运输挤压导致”；建议部分则给出“补发配件、申请理赔、安抚用户”等动作。

![售后处理流程](https://picsum.photos/seed/mock14/520/260)

| 信息来源 | 关键字段 | 摘要方式 | 是否可自动化 |
| --- | --- | --- | --- |
| 用户描述 | 问题、诉求、图片 | 提取核心诉求 | 是 |
| 物流轨迹 | 节点、签收、异常 | 判断运输风险 | 是 |
| 商品档案 | 品类、保修期、配件 | 判断售后政策 | 部分 |
| 历史记录 | 处理动作、赔付 | 避免重复处理 | 是 |

在界面上，可以把摘要放在工单顶部，但不要替代原始记录。客服需要随时展开原文，确认 AI 是否漏掉细节。对于高风险场景，例如大额退款、法律投诉、平台介入，AI 只能给建议，不能自动提交最终动作。参考 [售后自动化准则](https://example.com/docs/support-ai)。

```kotlin
data class TicketSummary(
    val facts: List<String>,
    val risk: String,
    val suggestedAction: String
)
```

好的售后摘要应该短、准、可追溯。它不追求写得像文章，而是帮助客服在十秒内理解当前工单为什么重要、下一步该做什么。

补充压测段落：在真实售后系统里，同一条 AI 回复可能继续追加处理记录、用户情绪判断和风险等级说明。页面需要支持用户快速滑过这些长内容，并在停留时完整渲染 Markdown。对于图片证据较多的工单，点击图片时仍要展示当前 Markdown 中全部图片地址，避免客服只看到局部证据。

性能观察补充：这类内容会同时触发普通段落、图片块、表格块和代码块渲染，适合验证列表快速滑动、停止滑动、回滑复用时是否发生重复解析或明显掉帧。

# 长文本会话 20：搜索结果排序

搜索排序需要平衡相关性、转化率、库存、价格、时效和业务策略。AI 解释排序结果时，不能只说“综合得分较高”，而要告诉运营每个因素如何影响排序。例如某商品相关性高但库存低，可以降低排序；某商品转化好但配送慢，也需要根据场景做权衡。

![搜索排序](https://picsum.photos/seed/mock20/520/260)

| 因素 | 影响方向 | 示例 | 可配置 |
| --- | --- | --- | --- |
| 文本相关性 | 正向 | 标题命中核心词 | 否 |
| 历史转化 | 正向 | 点击后购买率高 | 部分 |
| 库存状态 | 负向 | 即将售罄 | 是 |
| 配送时效 | 正向 | 本地仓次日达 | 是 |

排序系统建议提供可解释面板，展示查询词、召回来源、特征分和最终排序原因。这样当运营反馈“为什么这个商品排在前面”时，研发不需要临时查日志。更多内容见 [搜索排序说明](https://example.com/docs/search-ranking)。

```text
score = relevance * 0.50 + conversion * 0.25 + freshness * 0.10 + logistics * 0.15
```

最终排序不是固定公式，而是一套持续实验的系统。每次策略调整都需要记录实验版本、影响范围和核心指标，避免短期 GMV 提升掩盖长期体验下降。

补充压测段落：搜索排序解释可能包含召回链路、特征权重、实验分桶和业务干预记录。用户快速浏览历史对话时，不需要每条都马上变成完整富文本；但当停在当前 item 时，表格、链接和代码块必须完整可用。这条数据用于验证“全部加载数据，但按需渲染视图”的体验是否足够平滑。

性能观察补充：排序解释里的公式和表格对排版比较敏感，延迟渲染后仍需要检查高度变化是否自然，以及复用 item 时是否展示了上一条旧内容。

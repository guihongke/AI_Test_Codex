# 长文本会话 27：项目排期评审

项目排期评审的核心是识别依赖和风险，而不是把所有任务平均分给每个人。AI 可以根据需求描述生成任务拆解，再提示哪些部分需要提前验证，例如接口契约、数据迁移、兼容范围和灰度策略。排期中要明确不可压缩的时间，例如测试回归和应用商店审核。

![项目排期](https://picsum.photos/seed/mock27/520/260)

| 阶段 | 任务 | 风险 | 缓解 |
| --- | --- | --- | --- |
| 需求 | 明确边界 | 范围膨胀 | 冻结版本目标 |
| 设计 | 接口评审 | 字段反复 | 契约先行 |
| 开发 | 核心链路 | 依赖阻塞 | 提前 mock |
| 测试 | 回归验证 | 环境不稳 | 准备数据集 |

评审时建议把任务分成必须、可降级和可延期三类。这样遇到风险时，团队可以快速做取舍，而不是临近上线才整体延期。更多方法见 [项目排期指南](https://example.com/docs/project-plan)。

```text
MVP 范围：核心链路、基础异常、埋点。
延期范围：高级筛选、批量导出、个性化配置。
```

好的排期不是精确到每小时，而是对不确定性保持诚实。越早暴露风险，项目越容易稳定交付。

补充压测段落：项目排期讨论经常会继续追加角色分工、里程碑、风险矩阵和降级预案。Markdown 内容看起来只是文本，但表格、列表和代码块都会触发布局与测量。列表页如果一次性加载很多历史会话，需要把数据加载、布局占位和富文本渲染拆开处理，避免首屏变慢。

性能观察补充：排期内容常被反复上下查看，因此 holder 状态要记录清楚，避免同一条消息在复用后反复显示占位或重复执行完整 Markdown 渲染。

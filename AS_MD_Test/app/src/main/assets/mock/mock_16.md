# 长文本会话 16：财务对账异常

财务对账异常通常不是单点问题，而是订单、支付、退款、优惠和结算多个系统之间的时间差或口径差。AI 回复应该先解释异常类型，再给出排查路径。例如“支付成功但订单未确认”需要查支付回调；“退款成功但结算未扣减”需要查账期；“优惠承担方不一致”需要查活动配置。

![对账异常趋势](https://picsum.photos/seed/mock16/520/260)

| 异常类型 | 可能原因 | 排查入口 | 自动修复 |
| --- | --- | --- | --- |
| 长款 | 订单取消但支付入账 | 支付回调日志 | 部分 |
| 短款 | 订单确认但支付缺失 | 支付流水 | 否 |
| 退款差异 | 退款到账与结算账期不同 | 退款单 | 部分 |
| 优惠差异 | 平台券和商家券口径不同 | 活动配置 | 否 |

建议建立统一的对账事件模型，把订单号、支付单号、退款单号、结算单号和活动号串起来。端上展示时，先给出异常摘要，再展示流水链路。用户点击任一节点时，可以跳到对应系统的详情页。说明文档见 [对账事件模型](https://example.com/docs/reconcile)。

```yaml
event:
  orderId: O20260610001
  payId: P20260610001
  type: refund_gap
  amount: 18.90
```

财务页面要避免模糊措辞。每一个建议都应该附带金额、时间、单号和系统来源，这样财务同学才能快速判断是否需要人工介入。

补充压测段落：对账类 AI 回复经常包含多张表格、长编号和跨系统链接，这些内容会增加测量与排版成本。页面需要保证表格横向滑动时不影响外层列表，并且在列表快速滑动时不触发重复渲染。对于已经渲染过的 item，应尽量复用状态，减少从 Markwon 到 View 的重复转换。

性能观察补充：如果真实单条 Markdown 加载接近 0.6 秒，应优先减少滑动中的同步渲染，并把可见项渲染拆成多个小任务，避免连续多个 item 同帧阻塞。

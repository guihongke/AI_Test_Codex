# 长文本会话 15：内容审核队列

内容审核队列需要兼顾效率和谨慎。AI 可以先做风险预判，把明显安全的内容降权，把高风险内容提前，但最终是否封禁、限流或提醒，仍然需要结合业务政策和人工复核。队列设计上应当展示风险原因，而不只是一个分数，否则审核员很难信任模型结论。

![审核队列](https://picsum.photos/seed/mock15/520/260)

| 风险类型 | 示例信号 | 推荐处理 | 备注 |
| --- | --- | --- | --- |
| 广告引流 | 外部联系方式、重复文案 | 限流或人工复核 | 注意误伤品牌活动 |
| 低俗内容 | 敏感词、图片识别 | 人工复核 | 需要上下文 |
| 虚假信息 | 夸大承诺、伪造截图 | 下架并记录 | 保留证据 |
| 版权风险 | 搬运、无授权素材 | 提醒或下架 | 支持申诉 |

队列可以按照风险分、发布时间、传播速度和举报量综合排序。对于热点内容，传播速度比单条风险分更重要，因为处理延迟会带来更大扩散。审核结果需要写回样本库，用于后续模型评估和规则调优。更多策略见 [社区审核规范](https://example.com/docs/moderation)。

```sql
select content_id, risk_type, risk_score
from moderation_queue
where status = 'pending'
order by risk_score desc, report_count desc;
```

审核产品的体验重点是减少重复判断，让证据、政策和动作尽量在一个页面完成闭环。AI 的价值不是替代审核员，而是把注意力推向真正需要判断的内容。

补充压测段落：如果审核队列很长，用户会频繁上下滑动寻找重点内容。此时不适合在滑动过程中同步构建复杂 Markdown，否则会出现明显卡顿。更好的做法是在滚动时展示轻量占位或已缓存内容，等列表空闲后再渲染当前可见项。这条 mock 用来验证这种延迟渲染策略是否自然。

性能观察补充：审核场景经常有长文本证据和策略链接，停止滑动后再渲染不会影响业务数据加载完成状态，但能把重活避开手势过程。

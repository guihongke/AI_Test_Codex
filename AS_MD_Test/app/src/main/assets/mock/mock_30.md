# 长文本会话 30：AI 提示词治理

提示词治理的重点是版本、评估和回滚。随着业务复杂度提高，提示词会变成真正的生产配置。AI 回复质量不稳定时，需要知道当前使用的是哪个版本、改了什么、影响哪些场景，以及有没有通过评估集。不要把提示词散落在代码和后台配置里。

![提示词治理](https://picsum.photos/seed/mock30/520/260)

| 模块 | 治理内容 | 风险 | 建议 |
| --- | --- | --- | --- |
| 版本 | prompt 变更记录 | 无法回溯 | Git 化管理 |
| 评估 | 场景测试集 | 主观判断 | 自动评分加人工抽检 |
| 灰度 | 小流量验证 | 全量事故 | 分人群发布 |
| 回滚 | 快速恢复 | 配置污染 | 保留稳定版本 |

提示词上线前建议跑固定评估集，包括正常问题、边界问题、攻击输入和空数据场景。输出格式也要校验，尤其是需要 JSON 或结构化动作时。参考 [提示词发布流程](https://example.com/docs/prompt-release)。

```json
{
  "promptVersion": "support_summary_v12",
  "evalScore": 0.91,
  "rollbackTo": "support_summary_v11"
}
```

提示词不是一次性文案，而是可迭代的软件资产。把它纳入工程流程，才能让 AI 能力稳定进入生产系统。

补充压测段落：提示词治理回复常包含版本表、评估 JSON、失败案例和上线 checklist。真实项目中，这些内容可能远超过一千字符。渲染优化不能只依赖更快的机器，而应该减少不必要的同步工作。列表滑动时展示占位，停止后渲染可见项，是兼容 ListView 的保守方案。

性能观察补充：如果未来允许用户复制整段提示词，需要继续增强跨块选择能力；当前压测重点仍是长 Markdown 首屏加载和滑动卡顿。

# 长文本会话 31：报表导出中心

报表导出中心需要处理权限、异步任务、文件保存、过期清理和失败重试。AI 可以帮助用户生成报表说明，也可以在导出失败时解释原因。对于大数据量报表，端上不应该等待同步接口返回文件，而应该创建任务，然后通过任务列表查看进度。

![报表导出](https://picsum.photos/seed/mock31/520/260)

| 状态 | 含义 | 用户动作 | 系统动作 |
| --- | --- | --- | --- |
| 排队中 | 等待执行 | 可取消 | 调度任务 |
| 生成中 | 正在查询和写文件 | 查看进度 | 分片写入 |
| 已完成 | 文件可下载 | 下载或分享 | 设置过期时间 |
| 失败 | 任务异常 | 重试或反馈 | 记录错误 |

导出任务应该保存筛选条件、创建人、权限快照和文件过期时间。下载时仍需校验权限，不能因为任务创建成功就永久可访问。更多设计见 [导出中心方案](https://example.com/docs/export-center)。

```go
type ExportTask struct {
    ID string
    Status string
    Creator string
    ExpireAt int64
}
```

报表导出的体验重点是可预期。用户需要知道任务排队多久、失败能否重试、文件保留多久，以及导出的数据口径是否和页面一致。

补充压测段落：导出中心回复可能继续描述权限快照、任务调度、失败重试、文件清理和审计日志。因为历史任务多、消息长、表格多，首屏加载和滚动性能很容易受到 Markdown 解析影响。当前 mock 用来模拟这种压力，并验证延迟渲染不会影响最终图片、链接、表格和代码显示。

性能观察补充：导出任务列表通常还会关联下载地址和失败日志，链接点击弹窗、图片弹窗和表格横滑都需要在完整渲染后继续保持可用。

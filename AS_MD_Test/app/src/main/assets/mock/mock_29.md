# 长文本会话 29：数据大屏解读

数据大屏不应该只是把图表堆满屏幕。AI 解读可以帮助用户快速知道“发生了什么、为什么、要不要处理”。对于实时大屏，异常解释比静态指标更重要。页面上可以保留核心数字，但要把同比、环比、阈值和业务事件一起展示。

![数据大屏](https://picsum.photos/seed/mock29/520/260)

| 指标 | 当前值 | 变化 | 解释 |
| --- | ---: | ---: | --- |
| GMV | 1,280 万 | +9% | 活动流量提升 |
| 支付转化 | 32.4% | -1.8% | 移动端下降 |
| 客服排队 | 186 | +42 | 售后咨询增加 |
| 履约超时 | 74 | +19 | 华东仓拥堵 |

大屏解读最好关联业务事件，例如活动开始、系统发布、天气变化或供应链异常。没有事件解释的波动，很容易让用户误判。参考 [实时指标解读](https://example.com/docs/realtime-bi)。

```text
异常摘要：支付转化下降主要集中在 Android 5.8.1，建议检查最近发布。
```

大屏的价值是共识，而不是装饰。每个模块都应该回答一个业务问题，否则即使视觉很炫，也会降低决策效率。

补充压测段落：数据大屏解读通常会连续输出多段异常解释和处理建议。对于移动端对话列表，这类长回复如果同步渲染，会在快速滚动时非常明显地卡住。将 Markdown 渲染延迟到列表空闲后执行，可以把卡顿从手势过程中移开，同时保持最终展示效果完整。

性能观察补充：实时指标会频繁刷新，真实项目中还需要避免新数据到达时强制重渲染不可见 item，只更新模型，等 item 可见后再绑定。

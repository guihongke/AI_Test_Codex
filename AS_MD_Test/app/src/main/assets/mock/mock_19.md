# 长文本会话 19：数据埋点校验

埋点校验的目标不是让事件数量越多越好，而是让关键链路可观测、口径可复用、问题可追踪。AI 可以根据页面结构和业务目标生成埋点清单，但上线前仍需要研发、数据和产品共同确认字段含义。尤其是金额、用户状态和实验分组字段，必须保持稳定。

![埋点链路](https://picsum.photos/seed/mock19/520/260)

| 页面 | 事件 | 必填字段 | 校验方式 |
| --- | --- | --- | --- |
| 商品详情 | view_product | product_id, source | 实时日志 |
| 加购弹窗 | add_to_cart | sku_id, quantity | 端到端回放 |
| 结算页 | submit_order | order_id, amount | 服务端比对 |
| 支付页 | pay_result | pay_id, status | 支付流水核对 |

推荐建立埋点 schema 仓库，所有事件都走评审和版本管理。端上 SDK 可以在 debug 环境中校验字段缺失、类型错误和枚举越界。对于核心漏斗，最好同时有客户端事件和服务端事实表做交叉校验。参考 [埋点治理规范](https://example.com/docs/tracking)。

```json
{
  "event": "submit_order",
  "required": ["order_id", "amount", "currency"],
  "version": 3
}
```

埋点文档要服务于长期维护。字段名、触发时机、业务含义和示例值都应该写清楚，否则几个月后很容易出现同名不同义或同义不同名的问题。

补充压测段落：埋点类回复中经常出现 JSON、表格和链接，尤其适合测试 MarkdownView 的混排能力。滑动列表时，未停留的 item 没必要立即完成高成本渲染；当用户停在某条消息上时，再渲染完整格式。这样的策略在实际项目中可以明显减少首屏加载和连续滑动的压力。

性能观察补充：埋点字段较多时，表格会变宽，横向滑动只应发生在表格内部，外层 ListView 仍保持纵向滚动优先。

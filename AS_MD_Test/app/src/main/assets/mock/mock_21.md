# 长文本会话 21：客服知识库召回

知识库召回的关键是让答案既准确又可追溯。AI 可以把用户问题改写成多个检索查询，再从 FAQ、操作手册、政策文档和历史工单中召回候选答案。最终回复必须带来源，否则客服无法判断答案是否可靠。对于政策类问题，过期文档应该直接降权或禁止召回。

![知识库召回](https://picsum.photos/seed/mock21/520/260)

| 来源 | 优点 | 风险 | 处理建议 |
| --- | --- | --- | --- |
| FAQ | 简短稳定 | 覆盖有限 | 作为首选来源 |
| 操作手册 | 步骤详细 | 更新滞后 | 检查版本 |
| 历史工单 | 贴近真实问题 | 可能有错误处理 | 仅作参考 |
| 政策文档 | 权威 | 条款复杂 | 保留原文链接 |

召回后可以用重排模型评估问题和候选段落的匹配度，再让大模型生成答案。生成时应尽量引用原文术语，不要随意扩写政策。参考 [知识库 RAG 指南](https://example.com/docs/rag-support)。

```java
List<Document> docs = retriever.search(rewrittenQueries);
Answer answer = generator.generate(question, docs);
```

好的客服知识库不是把文档塞给模型就结束，而是需要文档治理、版本控制、召回评估和人工反馈闭环共同支撑。

补充压测段落：RAG 场景的回复通常还会列出引用片段、相似问题、命中文档版本和不确定性说明。这些内容对客服很有价值，但也会让单条消息更长。渲染层需要把“数据已经加载完成”和“富文本已经渲染完成”区分开，前者影响业务状态，后者只影响当前可见区域展示。

性能观察补充：引用内容越多，TextView 选择和链接点击越容易互相影响，因此停留渲染后还要验证链接弹窗、图片弹窗和复制菜单是否仍正常。

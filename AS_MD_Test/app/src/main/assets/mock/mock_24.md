# 长文本会话 24：风控规则解释

风控规则解释需要在安全和可理解之间取得平衡。对内部运营，可以展示较完整的命中规则、证据和历史行为；对外部用户，只能展示必要原因，避免暴露规则细节导致绕过。AI 可以帮助生成审核摘要，但不能泄露敏感阈值。

![风控规则](https://picsum.photos/seed/mock24/520/260)

| 规则组 | 命中信号 | 处理动作 | 是否可对外展示 |
| --- | --- | --- | --- |
| 账号异常 | 设备频繁切换 | 二次验证 | 部分 |
| 交易异常 | 短时高频下单 | 延迟审核 | 否 |
| 优惠滥用 | 多账号同设备 | 限制领券 | 部分 |
| 内容风险 | 敏感词组合 | 人工复核 | 部分 |

内部页面建议展示规则链路：原始事件、特征计算、规则命中、模型评分和最终动作。这样当用户申诉时，运营可以快速判断是否误判。策略说明见 [风控解释规范](https://example.com/docs/risk-explain)。

```json
{
  "riskLevel": "medium",
  "action": "verify",
  "explain": ["device_changed", "coupon_frequency_high"]
}
```

风控系统不能只追求拦截率，也要关注误伤率和申诉通过率。解释能力越清楚，后续规则迭代就越稳。

补充压测段落：风控解释通常包含敏感信号、内部备注和申诉处理建议。UI 层不能因为内容长就阻塞用户滚动，也不能在输出过程中误触链接或图片。当前 mock 用于验证静态全量加载模式下，长 Markdown、表格、图片和链接是否能在停留后完整渲染，并保持后续交互可用。

性能观察补充：风控内容的表格列名可能很长，应确认列宽预估不会频繁变化，并且横滑手势不会被外层 ListView 抢走。

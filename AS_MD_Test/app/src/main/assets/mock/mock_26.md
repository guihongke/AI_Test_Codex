# 长文本会话 26：App 崩溃分析

崩溃分析需要先判断影响范围，再定位版本、机型、系统和页面路径。AI 可以把堆栈聚类成问题摘要，并关联最近发布、灰度配置和实验开关。对于高频崩溃，页面应该优先展示复现路径和规避方案，而不是让研发自己翻大量日志。

![崩溃趋势](https://picsum.photos/seed/mock26/520/260)

| 维度 | 异常表现 | 可能原因 | 动作 |
| --- | --- | --- | --- |
| 版本 | 5.8.1 激增 | 新增模块空指针 | 暂停灰度 |
| 机型 | 某品牌集中 | 系统 WebView 差异 | 兼容处理 |
| 页面 | 订单详情 | 字段缺失 | 加兜底 |
| 实验 | B 组明显更高 | 新布局问题 | 关闭实验 |

聚类时应保留原始堆栈、发生次数、影响用户数和首次出现时间。AI 生成结论时必须标记置信度，避免把相关性误当因果。参考 [移动端稳定性治理](https://example.com/docs/mobile-stability)。

```java
if (order == null) {

}
```

崩溃治理的优先级不只看次数，也要看是否阻断核心链路。支付页低频崩溃可能比普通设置页高频崩溃更值得优先处理。

补充压测段落：稳定性分析回复中经常包含堆栈、版本、机型、实验和修复建议。代码块和表格会增加 Markwon 渲染成本，图片又会引入异步资源处理。通过这一类长消息，可以更接近实际项目中的性能压力，验证滑动中暂停渲染是否能减少掉帧。

性能观察补充：崩溃堆栈可能非常长，后续如果继续扩展，应考虑代码块折叠或摘要优先显示，避免单条 item 高度过大影响列表定位。

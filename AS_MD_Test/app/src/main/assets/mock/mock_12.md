# 长文本会话 12：订单履约看板优化

这条 mock 用来模拟实际 AI 回复中的长段落、表格、图片和链接混合内容。系统建议把订单履约拆成三个层级：入口汇总、异常定位和动作执行。入口层只展示最重要的指标，例如今日待出库、超时风险、客服催单和仓库处理中；异常定位层按仓库、承运商、订单类型和时效等级继续下钻；动作执行层则提供批量改派、优先拣货、通知客服和生成补偿单等操作。这样的结构可以减少运营同学来回切换页面的次数，也能避免所有指标平铺后造成注意力分散。

![履约趋势图](https://picsum.photos/seed/mock12/520/260)

| 模块 | 当前问题 | 建议动作 | 优先级 |
| --- | --- | --- | --- |
| 仓库出库 | 高峰期批次等待时间偏长 | 增加波次预估并提前锁库存 | P0 |
| 承运商 | 部分区域揽收延迟 | 根据历史 SLA 自动改派 | P1 |
| 客服协同 | 催单信息不完整 | 在工单中追加履约节点摘要 | P1 |
| 监控告警 | 指标口径不统一 | 建立统一的异常状态枚举 | P2 |

实现时可以先建立一个 `FulfillmentSnapshot` 数据对象，后端聚合各系统状态后返回给端上。端上只负责展示、筛选和动作触发，避免在页面层拼装复杂业务逻辑。更多指标说明可以参考 [履约指标文档](https://example.com/docs/fulfillment)。如果要继续扩展，可以把风险订单按照“已超时、即将超时、承运商异常、库存异常”四类展示，并给每一类配置默认处理动作。

```java
class FulfillmentSnapshot {
    String warehouseId;
    int pendingCount;
    int riskCount;
    String nextAction;
}
```

最后要注意，运营页面不是报表页面，最关键的是让用户快速发现问题并完成处理。指标可以多，但首屏必须克制，默认排序应该优先展示“需要马上处理”的事项。

补充压测段落：真实对话里，AI 往往还会继续解释边界条件、失败兜底和后续迭代计划。例如当仓库系统短暂不可用时，页面应保留最近一次快照，并明确标记数据时间；当某个动作批量执行失败时，需要展示成功数、失败数和失败原因。列表滑动压测时，这类长段落可以验证 MarkdownView 在多段文本、表格、图片和代码混排下是否仍能保持稳定。

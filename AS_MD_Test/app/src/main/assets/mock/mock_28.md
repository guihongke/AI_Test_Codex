# 长文本会话 28：权限系统设计

权限系统要同时支持角色授权、资源范围和操作粒度。简单的 RBAC 在早期足够，但当组织层级复杂后，需要增加数据范围、临时授权和审计能力。AI 可以帮助梳理权限矩阵，但最终必须由业务负责人确认，避免默认给过大权限。

![权限矩阵](https://picsum.photos/seed/mock28/520/260)

| 角色 | 可见范围 | 可执行动作 | 审计要求 |
| --- | --- | --- | --- |
| 管理员 | 全部组织 | 配置、审批 | 强审计 |
| 区域经理 | 所属区域 | 查看、导出 | 普通审计 |
| 门店店长 | 单门店 | 查看、处理 | 普通审计 |
| 临时协作 | 指定资源 | 只读 | 到期回收 |

权限判断建议放在服务端，端上只做展示控制。所有敏感操作都应该由服务端二次校验，不能依赖按钮隐藏。审计日志至少记录操作者、资源、动作、时间和来源 IP。参考 [权限系统设计](https://example.com/docs/permission)。

```sql
select *
from permission_policy
where role_id = ? and action = ? and resource_type = ?;
```

权限设计最容易低估的是后期维护成本。命名、继承、过期和审计从一开始就要规范，否则系统规模变大后很难治理。

补充压测段落：权限系统回复可能包含大量角色矩阵和 SQL 示例。用户快速滑动查找某个角色时，最重要的是列表响应速度；当停在某条消息后，再展示完整表格和代码即可。这个 mock 主要用于测试重复进入、快速滑动、停留渲染和再次滑动时的状态切换。

性能观察补充：权限矩阵列数容易增加，只有表格内部可以横向滑动；普通段落仍应跟随外层列表纵向滚动，避免用户手势预期混乱。

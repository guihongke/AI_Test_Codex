# VSmartRefreshLayout 组件介绍

## 1. 组件定位

`refresh-layout` 是基于 SmartRefreshLayout 2.1.1 AndroidX 源码整理的本地刷新组件。
公开包名统一为：

```text
com.vivo.chat.weight.refresh.smartrefresh
```

公开类型统一增加 `V` 前缀，例如 `VSmartRefreshLayout`、`VRefreshLayout`、
`VClassicsHeader`。组件运行环境为 `minSdk 23`、`targetSdk 34`、Java 11。

本项目在原有刷新框架能力上增加了两个直接面向业务的能力：

1. `VRefreshRecyclerLayout`：为 RecyclerView 提供类似
   `ListView.addHeaderView(View)` 的 Header API。
2. `VProgressHeader` / `VProgressFooter`：刷新视图完全使用 XML 子 View 定义，
   适合放置项目自己的 ProgressBar 或状态布局。

## 2. 工程结构

| 模块或目录 | 作用 |
| --- | --- |
| `app` | 可运行 Demo，包含 XML Progress、Classics、Material 三种样式 |
| `refresh-layout` | Android Library 源码和资源 |
| `refresh-layout/src/main/java/.../api` | Layout、Header、Footer、Kernel 等接口 |
| `.../header`、`.../footer` | 内置刷新组件 |
| `.../listener` | 刷新、加载、状态和边界监听器 |
| `.../constant` | `VRefreshState`、`VSpinnerStyle` 等状态常量 |

## 3. Layout 的三角色设计

`VSmartRefreshLayout` 是一个最多包含三个角色的 `ViewGroup`：

```text
VSmartRefreshLayout
├── Refresh Header：下拉时显示
├── Content：RecyclerView、NestedScrollView、普通 View 等业务内容
└── Refresh Footer：上拉时显示
```

XML 中最多放三个直接子 View。推荐顺序始终是 Header、Content、Footer。框架在
`onFinishInflate()` 中识别可滚动 Content；实现了 `VRefreshHeader` 或
`VRefreshFooter` 的子 View 会直接作为刷新组件，普通 View 位于 Header/Footer 位置时
会由 Wrapper 包装。缺少 Content 属于布局错误，运行时会放置错误提示 View。

注意以下两个 Header 含义不同：

| 名称 | 归属 | 行为 |
| --- | --- | --- |
| Refresh Header | `VSmartRefreshLayout` | 下拉时出现，刷新结束后收起 |
| RecyclerView Header | RecyclerView Adapter | 是列表第一条内容，跟随列表滚动 |

## 4. VRefreshRecyclerLayout 的 Header 设计

`VRefreshRecyclerLayout` 继承 `VSmartRefreshLayout`，内部使用 `ConcatAdapter`：

```text
RecyclerView.Adapter
├── VHeaderViewAdapter：0..N 个 Header View
└── 业务 Adapter：真实数据
```

因此 Header 不会侵入业务 Adapter，也不会要求业务 Adapter 预留 viewType。实现还处理了：

- 同一个 View 实例不会重复添加。
- 删除 Header 后使用局部 `notifyItemRemoved`。
- `GridLayoutManager` 下 Header 自动占满所有列，并保留业务原有
  `SpanSizeLookup`。
- `StaggeredGridLayoutManager` 下 Header 自动设置 `fullSpan=true`。
- Header 被回收或复用时会先从旧父容器移除，避免 `View already has a parent`。

业务必须调用 `VRefreshRecyclerLayout.setAdapter()`。若直接调用
`getRecyclerView().setAdapter()`，内部 Header Adapter 不会被拼接，之后调用
`addHeaderView()` 也不会显示。

## 5. 拖动、阻尼与回弹设计

手指的真实移动距离不会一比一映射到 Content 偏移。核心参数如下：

| 参数 | 默认值 | 含义 |
| --- | ---: | --- |
| `dragRate` | `0.5` | View 偏移量相对手指移动量的比例，越小阻尼越强 |
| `headerTriggerRate` | `1.0` | 触发刷新距离 / Header 高度 |
| `footerTriggerRate` | `1.0` | 触发加载距离 / Footer 高度 |
| `headerMaxDragRate` | `2.5` | Header 最大拖动距离 / Header 高度 |
| `footerMaxDragRate` | `2.5` | Footer 最大拖动距离 / Footer 高度 |
| `reboundDuration` | `300ms` | 放手回弹和完成收起动画的基础时长 |

`setDragRate(0.35f)` 表示手指移动 100px 时，初始可见偏移大约为 35px；超过
Header/Footer 基准高度后还会进入非线性衰减区域，所以实际越往外拉越重。

阻尼和回弹是两个不同阶段：

1. 手指按住移动：由 `dragRate`、最大拖动距离和非线性衰减共同决定。
2. 手指松开：由状态机决定回到 0，或回到 Header/Footer 工作高度；动画由
   `reboundDuration` 和 `Interpolator` 决定。

合理的业务配置通常为：

```java
refreshLayout
        .setDragRate(0.45f)
        .setHeaderTriggerRate(1.0f)
        .setFooterTriggerRate(1.0f)
        .setHeaderMaxDragRate(2.0f)
        .setFooterMaxDragRate(2.0f)
        .setReboundDuration(300);
```

不建议把 `dragRate` 设为 0；用户手势将几乎看不到反馈。接近 1 时阻尼很弱，
短距离手势就会造成明显位移。Demo 提供 `0.1~1.0` 滑杆用于实时比较。

## 6. 状态机

常用 Header 流程：

```text
None
  -> PullDownToRefresh
  -> ReleaseToRefresh
  -> RefreshReleased
  -> Refreshing
  -> RefreshFinish
  -> None
```

未达到阈值松手时：

```text
PullDownToRefresh -> PullDownCanceled -> None
```

Footer 对应状态为 `PullUpToLoad`、`ReleaseToLoad`、`LoadReleased`、`Loading`、
`LoadFinish`。`VRefreshState` 还提供以下布尔字段用于避免硬编码枚举集合：

| 字段 | 含义 |
| --- | --- |
| `isHeader` / `isFooter` | 当前状态属于 Header 还是 Footer |
| `isDragging` | 手指正在拖动 |
| `isOpening` | 正在刷新、加载或二级展开 |
| `isFinishing` | 正在播放完成动画 |
| `isReleaseToOpening` | 当前松手会立即进入刷新/加载 |

业务通常只需要刷新和加载监听器；埋点、调试面板或复杂联动才需要
`VOnStateChangedListener` / `VOnMultiListener`。

## 7. Header/Footer 生命周期

自定义刷新组件实现 `VRefreshHeader` 或 `VRefreshFooter`，主要生命周期为：

| 回调 | 调用时机 |
| --- | --- |
| `onInitialized` | Header/Footer 高度和最大拖动距离确定后 |
| `onMoving` | 手指拖动以及回弹动画每一帧 |
| `onReleased` | 达到触发条件后松手 |
| `onStartAnimator` | 正式开始刷新或加载动画 |
| `onFinish` | 业务调用 finish 后；返回值是完成状态额外停留时间 |
| `onStateChanged` | 状态机发生变化 |
| `setPrimaryColors` | Layout 向组件分发主题色 |

`VSpinnerStyle` 决定组件如何参与布局和位移：

| 类型 | 行为 |
| --- | --- |
| `Translate` | Header/Footer 自身跟随偏移平移，最常用 |
| `Scale` | 高度随拖动距离拉伸 |
| `FixedBehind` | 固定在 Content 后方，由 Content 移开后露出 |
| `FixedFront` | 固定在 Content 前方 |
| `MatchLayout` | 占满 Layout，适合特殊全屏交互 |

`VProgressHeader` 和 `VProgressFooter` 已实现生命周期，业务只需在 XML 中提供可见
子 View。`VProgressFooter` 在无更多数据时会递归隐藏内部 `ProgressBar`。

## 8. Content 与滚动边界

Content 由内部 `VRefreshContentWrapper` 适配。框架可识别 RecyclerView、
NestedScrollView、ViewPager/ViewPager2、CoordinatorLayout 等常见 AndroidX 容器。

开始下拉前必须满足 `canRefresh(content)=true`，开始上拉前必须满足
`canLoadMore(content)=true`。复杂页面可用 `setScrollBoundaryDecider()` 覆盖判定，
例如只允许某个 Tab 或某个子列表响应刷新。

嵌套滚动用于接收 RecyclerView 等 NestedScrollingChild 的未消费垂直距离。直接触摸
和嵌套滚动最终都会进入同一 spinner 偏移、状态机和回弹路径，避免两套动画行为不一致。

## 9. 无更多数据设计

成功加载最后一页时推荐一次性调用：

```java
refreshLayout.finishLoadMore(0, true, true);
```

第三个参数会在加载完成动画的正确时机进入 no-more-data 状态。刷新第一页成功后应调用：

```java
refreshLayout.resetNoMoreData();
```

`setNoMoreData(boolean)` 适合确实需要立即改变状态的场景；普通分页流程优先使用
`finishLoadMore(..., noMoreData)`、`finishLoadMoreWithNoMoreData()` 和
`resetNoMoreData()`，动画与状态更同步。

## 10. 全局默认组件

应用可在 `Application.onCreate()` 中设置全局初始化器和默认 Header/Footer Creator。
Layout 自身没有声明对应组件时才会使用 Creator：

```java
VSmartRefreshLayout.setDefaultRefreshInitializer((context, layout) ->
        layout.setDragRate(0.45f).setReboundDuration(300));

VSmartRefreshLayout.setDefaultRefreshHeaderCreator((context, layout) ->
        new VClassicsHeader(context));

VSmartRefreshLayout.setDefaultRefreshFooterCreator((context, layout) ->
        new VClassicsFooter(context));
```

全局设置影响进程中之后创建的 Layout，测试用例和多业务模块应避免在不受控位置反复覆盖。

## 11. 兼容性边界

- 工程使用 AndroidX，不应与旧 support library 混用。
- 最低 Android 版本为 API 23，编译和目标版本为 API 34。
- RecyclerView Header 依赖 `androidx.recyclerview:recyclerview:1.3.2` 的
  `ConcatAdapter`。
- 组件处理触摸、NestedScrolling、RecyclerView/Grid/StaggeredGrid 等主路径；
  WebView、地图、横向 ViewPager 嵌套等复杂手势页面仍应在目标机型上专项验证边界判定。
- 本地分支保留 Apache 2.0 `LICENSE` 和 `NOTICE`，AAR assets 中也包含对应文件。

具体接入代码、全部业务 API 和 XML 属性见项目根目录的《使用文档.md》。
